# 实时行情数据采集平台 — 数据层需求文档（tz-data）

> **所属工程**：tz-data (`C:\myspace\tz-data`)
> **原则**：tz-data 只负责数据采集、存储、质量监控。所有面向用户的展示、分析、交互需求归属 tz2.0。
> **参考**：`C:\myspace\tz2.0\docs\hq\实时行情需求.md`（总体架构设计）

## 一、职责范围

| 职责 | 说明 |
|------|------|
| **数据采集** | 从多种数据源（CTP/iTick/Tushare/AKShare/腾讯财经）获取实时与历史行情 |
| **数据存储** | QuestDB（热数据）、Redis（快照）、SQLite（元数据/日志）、Parquet（冷数据归档） |
| **质量监控** | 延迟检测、完整性校验、合理性检查、缺口回补、告警分级 |
| **目录管理** | 维护 `market_data_catalog` 表，控制订阅范围和数据源路由 |
| **事件记录** | 结构化记录所有生命周期事件（连接、断开、重连、切换、回补等） |
| **定时任务** | 盘前快照、缺口检测、质量日报、合约到期清理、数据归档 |

## 二、数据源驱动层

### 2.1 统一驱动接口

新建 `src/tzdata_pkg/market/drivers/base_driver.py`，定义 `BaseMarketDriver` 抽象基类：

```python
class BaseMarketDriver(ABC):
    SOURCE_NAME: str = "base"        # 数据源唯一标识
    
    async def connect(self, config: dict) -> None: ...
    async def subscribe(self, symbols: list[str]) -> None: ...
    async def unsubscribe(self, symbols: list[str]) -> None: ...
    def on_data(self, callback: Callable[[dict], None]) -> None: ...
    async def heartbeat(self) -> dict: ...
    async def disconnect(self) -> None: ...
```

**设计说明**：
- 与现有 `BaseDataSource`（用于盘后数据源）和 `BaseExchangeDownloader`（用于下载器）不同，`BaseMarketDriver` 专为**实时流式**数据源设计
- 驱动层负责将原始数据转换为 `UnifiedMarketData` 格式后触发回调

### 2.2 分阶段驱动接入

| 阶段 | 驱动 | 文件 | 延迟 | 接入方式 |
|------|------|------|------|----------|
| Phase 1 | Tushare | `drivers/tushare_driver.py` | 分钟~日级 | HTTP 轮询 |
| Phase 1 | AKShare | `drivers/akshare_driver.py` | 分钟~日级 | HTTP 轮询 |
| Phase 1 | 腾讯财经 | `drivers/qq_finance_driver.py` | 1~3 秒 | HTTP 轮询（过渡方案） |
| Phase 2 | CTP | `drivers/ctp_driver.py` | Tick 级 (<10ms) | 事件驱动（CTP MdApi） |
| Phase 3 | iTick | `drivers/itick_driver.py` | 0.5~3 秒 | SDK/WS 接口 |

**Phase 1 优先级最高**：先实现轮询式驱动，验证适配层框架。CTP 驱动依赖 CTP Python SDK 可用性。

### 2.3 驱动扩展性

新增数据源只需：
1. 继承 `BaseMarketDriver`
2. 实现 6 个抽象方法
3. 在配置中注册，适配层动态加载

## 三、统一数据模型

新建 `src/tzdata_pkg/market/models.py`，定义 `UnifiedMarketData`：

```python
@dataclass
class UnifiedMarketData:
    # 通用标识
    symbol: str              # "MO2406C3900"
    exchange: str            # "CFFEX"
    asset_type: str          # "FUTURE" / "OPTION" / "INDEX"
    timestamp: int           # 毫秒级 Unix 时间戳

    # 通用行情
    open: float | None
    high: float | None
    low: float | None
    close: float | None
    volume: int | None
    open_interest: int | None
    pre_close: float | None
    bid_price1: float | None
    bid_volume1: int | None
    ask_price1: float | None
    ask_volume1: int | None

    # 期权结构性字段（期权必填，其他为空）
    option_type: str | None          # "CALL" / "PUT"
    strike_price: float | None
    expiry_date: str | None
    underlying_symbol: str | None
    underlying_price: float | None

    # 期权估值与风险
    implied_volatility: float | None
    delta: float | None
    gamma: float | None
    vega: float | None
    theta: float | None
    rho: float | None

    # 计算辅助
    moneyness: str | None            # "ITM" / "ATM" / "OTM"
    days_to_expiry: int | None
    theoretical_price: float | None

    # 质量标记
    data_source: str                 # 实际数据来源 ("ctp", "tushare", ...)
    is_backfill: bool                # 是否为回补数据
    data_quality: str                # "normal" / "degraded" / "suspect"
```

**为什么统一而非分开**：避免上层应用维护两套代码，跨品种操作（如组合 Delta 汇总）自然高效。NULL 字段对 Parquet 列式存储几乎不占空间。

## 四、行情适配层

新建 `src/tzdata_pkg/market/adapter.py`，定义 `MarketDataAdapter` 核心中枢：

### 4.1 主题订阅管理
- 上层应用通过 WebSocket 订阅 symbol 列表
- 适配层维护订阅计数，计数从 0 变 1 时向驱动层发起采集，归零时停止
- 新订阅者上线时先推送 Redis 中的最新快照，再推送增量

### 4.2 数据归一化
- 所有驱动上报的数据在适配层转换为 `UnifiedMarketData`
- 字段名映射、单位转换、精度统一

### 4.3 多源路由与切换
- 根据 `market_data_catalog` 配置，优先从 `real_time_source` 获取数据
- 主源连续超时或错误超过阈值（5 秒无心跳），自动切换至 `backup_source`，并发送告警
- 切换后若主源恢复，自动切回

### 4.4 数据质量校验
- **时效性**：延迟超过阈值（实盘 2 秒，非实盘 10 秒）的数据丢弃并计数
- **完整性**：必填字段（OHLC、成交量）为空则丢弃
- **合理性**：价格超出涨跌停、IV 超出 (0, 200%)、Delta 超出 (-1, 1) 则标记 `data_quality='suspect'` 但仍推送
- **期权专项**：Call 的 Delta 应为正、Put 应为负，违反则标记

### 4.5 降级与熔断
- **降级触发**：所有实时源均不可用且持续 30 秒
- **降级行为**：停止订阅，改为每 5 秒从 Redis 推送最后一次有效快照，标记 `data_quality='degraded'`
- **自动恢复**：任一源连续正常推送 10 秒后退出降级
- **单 symbol 熔断**：某 symbol 连续 5 条异常数据则暂停推送并告警

### 4.6 缺口自动回补
- **检测**：扫描每个活跃 symbol，若交易时段内超过 5 秒无 tick，判定为缺口
- **回补**：向数据下载模块发起紧急补全，拉取缺失区间 K 线，标记 `is_backfill=true` 推送
- **记录**：缺口事件写入 `market_data_event_log`

### 4.7 多源时钟同步
- 统一时间基准为交易所时间
- 每日计算 CTP 时间与本地时钟的偏移量
- 非 CTP 源数据到达后，时间戳加上偏移量再处理

### 4.8 盘前开盘快照
- Celery Beat 定时任务，每日 09:25:05 执行
- 强制刷新所有活跃成分股和指数 symbol 的最新快照
- 以最高优先级推送给策略引擎

## 五、数据存储方案

### 5.1 SQLite 新表

在 `src/tzdata_pkg/storage/schemas/market.sql` 中新增：

**`market_data_catalog`** — 数据目录表

| 字段 | 类型 | 说明 |
|------|------|------|
| `symbol` | TEXT UNIQUE | 唯一标识，如 `IM2406`、`MO2406C3900` |
| `product_id` | TEXT | 所属品种，如 `IM_FUT`、`MO_OPT` |
| `exchange` | TEXT | 交易所，如 `CFFEX` |
| `asset_type` | TEXT | `FUTURE` / `OPTION` / `INDEX` |
| `is_active` | INTEGER | 是否启用采集 (1/0) |
| `real_time_source` | TEXT | 实时行情主数据源 |
| `backup_source` | TEXT | 实时行情备用数据源 |
| `historical_source` | TEXT | 历史数据源 |
| `subscribe_from` | TEXT | 自动订阅开始时间 |
| `subscribe_until` | TEXT | 自动订阅结束时间 |

**`data_source_status`** — 数据源状态表

| 字段 | 类型 | 说明 |
|------|------|------|
| `source_name` | TEXT UNIQUE | 数据源名称 |
| `source_type` | TEXT | `historical` / `realtime` / `transition` |
| `status` | TEXT | `connected` / `disconnected` / `degraded` / `error` |
| `last_heartbeat` | TIMESTAMP | 最后心跳时间 |
| `latency_ms` | INTEGER | P50 延迟 |
| `latency_p99_ms` | INTEGER | P99 延迟 |
| `error_count` | INTEGER | 累计错误数 |
| `symbols_subscribed` | INTEGER | 订阅 symbol 数 |

**`market_data_event_log`** — 事件日志表（保留 90 天）

| 字段 | 类型 | 说明 |
|------|------|------|
| `event_type` | TEXT | `connect` / `disconnect` / `reconnect` / `switch` / `backfill` / `gap` / `snapshot` |
| `source_name` | TEXT | 数据源名称 |
| `symbol` | TEXT | 相关 symbol |
| `severity` | TEXT | `info` / `warning` / `error` / `critical` |
| `message` | TEXT | 事件描述 |
| `details` | TEXT | JSON 详细信息 |

**`data_quality_metrics`** — 质量指标表

| 字段 | 类型 | 说明 |
|------|------|------|
| `symbol` | TEXT | symbol |
| `trade_date` | TEXT | 交易日 |
| `source_name` | TEXT | 数据源 |
| `delay_ms` | INTEGER | 平均延迟 |
| `gap_count` | INTEGER | 缺口次数 |
| `missing_bars` | INTEGER | 缺失 bar 数 |
| `suspect_count` | INTEGER | 异常标记数 |
| `quality_score` | REAL | 质量得分 (0-100) |

**`realtime_snapshots_cache`** — 快照缓存表（Redis 镜像）

| 字段 | 类型 | 说明 |
|------|------|------|
| `symbol` | TEXT PK | symbol |
| `data_json` | TEXT | JSON 序列化的 UnifiedMarketData |
| `source_name` | TEXT | 数据来源 |
| `updated_at` | TIMESTAMP | 更新时间 |

### 5.2 QuestDB

- 表名：`market_data_realtime`
- 主排序键：`timestamp`，次索引：`symbol`
- 写入方式：ILP (InfluxDB Line Protocol) 批量写入
- 保留策略：30 天自动归档为 Parquet

### 5.3 Redis

- Key 格式：`rt:{symbol}`
- Value：JSON 序列化的 `UnifiedMarketData`
- TTL：自适应（降级模式下延长）

### 5.4 Parquet 归档

- 目录结构：`/data/market/historical/{asset_type}/{year}/{month}/{symbol}.parquet`
- 归档任务：每日凌晨 Celery 将 30 天前数据从 QuestDB 导出

## 九、监控告警

### 9.1 告警分级

| 级别 | 触发条件 | 动作 |
|------|---------|------|
| P0-紧急 | CTP 中断超 10 秒，主备切换失败 | IM + 电话告警，策略挂起 |
| P1-严重 | 单 symbol 缺失超 30 秒 | IM 告警，相关策略暂停 |
| P2-警告 | 延迟突增、备用源数据质量低 | IM 静默通知，记录日志 |

### 9.2 质量看板指标

- 各数据源延迟折线图（P50/P99）
- 今日推送总量、缺失次数、回补次数
- 各 symbol 健康状态热力图（绿/黄/红）

## 十、与 tz2.0 协作方式

| 方式 | 用途 | 说明 |
|------|------|------|
| HTTP API | 元数据查询、状态查询 | tz2.0 的 `TzDataApiClient` 调用 `/api/v1/market/*` |
| Python import | 批量数据查询 | tz2.0 直接 import `tzdata_pkg` 内部函数 |
| 共享 Redis | 实时快照推送 | tz-data 写入，tz2.0 WebSocket 读取推送给前端 |
| 共享 SQLite | 只读访问 | tz2.0 通过 SQLAlchemy 引擎直读 tz-data 数据库文件 |
