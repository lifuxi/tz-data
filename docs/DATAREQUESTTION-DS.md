## 数据维护与同步系统设计方案（基于层级架构）

### 一、 数据源层——多源统一接入

**设计目标**：屏蔽不同数据源的差异，向上层提供统一的“品种-合约-数据维度”抽象。

- **交易所与第三方**：
  - 通过插件化的 `BaseDataSource` 子类接入。目前已规划：`TushareSource`、`CFFEXSource` (爬虫)、`SHFESource`、`WindSource` 等。
  - 每个数据源实例化后注册到 `SourceManager`，由它响应上层的“拉取”命令。
- **本地数据源**：
  - **数据库**：PostgreSQL（元数据、账单、任务等）和 ClickHouse（海量K线行情、机构持仓时序数据）。
  - **文件存储**：用于账单原始文件（CSV/Excel）的留存，便于重新解析或审计。
- **账单源配置管理**：
  - **期货公司账户管理**：在“数据中心→账单Tab→设置”中配置账户，支持多账户。
  - **登录凭证安全存储**：CTP/监控平台密码经加密后存入`credential_vault`表。
  - **账单格式适配器**：一个内置的解析器注册表，能够根据账单文件名或内容自动识别（监控中心csv、博易大师、快期等格式），调用对应的解析逻辑。

---

### 二、 数据管理层——元数据与状态监控

**设计目标**：让系统清楚地“知道”自己应该有什么、实际有什么，这是整个数据维护中心的核心大脑。

#### 2.1 元数据管理（数据中心的基础配置入口）
- **交易所管理**：通过 `product_config` 表管理交易所启停。在Web界面“系统设置→数据源管理”中操作。
- **品种管理**：展现各交易所下挂靠的品种（如中金所→IM/IF/HO）；在该页面可以开启/关闭“跟踪”。
- **合约管理**：展示合约列表（自动发现+手动添加）。合约有状态：`活跃`、`到期`、`跟踪`。用户可手动调整跟踪状态。

#### 2.2 数据完整性检查
- **自动对账引擎**：一个Celery定时任务，每日凌晨执行，对比 `trade_calendar`（基准日历）与数据库中各类数据的实际日期：
  - **日线/分钟线**：检查每个被跟踪合约的K线覆盖日期。
  - **机构持仓**：检查 `cffex_top20_holdings` 表每日是否都有记录。
  - **账单数据**：检查用户设定的“跟踪起始日”以来每个交易日是否有解析成功的账单。
- **缺口生成**：差异结果存入 `data_health_snapshot` 表，作为前端数据浏览器快速展示的依据。

#### 2.3 数据质量评估与诊断（之前讨论的诊断引擎深入实现）
- **数据质量字段**：在快照表中增加 `data_quality_score`，由多因子加权：
  - 完整性（是否有缺失交易日）。
  - 准确性（多数据源交叉校验后的偏离率，来自DQ系列需求）。
  - 一致性（分钟数据汇总后与日线数据是否吻合）。
- **账单独有质量检查**：
  - 解析时自动执行“内部试算平衡”（如期末权益=期初+入金-出金+盈亏-手续费）。
  - 若不平，标记“数据存疑”，在数据中心账单列表中显示黄色警告。
- **差异对比引擎**：应用需求文档DQ-01~03，对同一品种的Tushare来源和官方来源数据进行收盘价对比，结果存入 `data_diff_log`，在数据看板展示。

### 三、 任务调度层——同步与修复的执行

**设计目标**：对上层“补齐”指令进行任务编排与执行。

- **同步模式**：
  - **增量同步**：日常自动拉取。对比本地最新日期与源最新日期，拉取差值。这是`每日自动对账`发现小缺口后的默认修复方式。
  - **全量同步**：手动触发，常用于数据源初始化或数据错乱后的整表重构。
  - **断点续传**：对于大批量下载（如补3个月分钟线），任务记录当前进度，失败重跑时从上次中断处继续，不重复拉取。
- **账单自动化同步**：
  - 通过监控平台API或目录监控，尽可能自动化获取账单（P3目标）。
  - 获取后自动触发解析工作流：`源文件→解析引擎→数据质量校验→入库→标记订单配对`。
  - 若解析失败，触发`缺失账单检测`，在数据中心产生一条待处理记录，推送IM通知。
- **手动任务触发**：
  - 用户在Web界面点击“一键补齐”后，接口会向任务队列（Celery）投递一个优先修复任务。
  - **账单补全工作流**：当用户批量上传缺失账单文件时，系统创建`BATCH_UPLOAD`任务，进入“解析→校验→报告”的完整流水线。

### 四、 存储层——物理数据落地

- **关系数据库（PostgreSQL）**：作为**元数据**（配置、账户、产品）、**任务管理**（调度日志、任务状态）、**状态记录**（Data Health Snapshot）的唯一事实来源。存储所有账单的交易明细、资金流水。
- **时序数据库（QuestDB / ClickHouse）**：存储所有**海量时间序列数据**，包括Tushare/CFFEX下载的全部分钟K线、日K线、机构持仓历史排名。保证回测和大数据量图表加载的极致性能。
- **缓存层（Redis）**：
  - **任务状态缓存**：异步下载/解析任务进度实时展示。
  - **数据状态缓存**：缓存主页“综合看板”的数据库健康状态，每5分钟刷新一次，确保仪表盘秒开。

### 五、 应用层——Web管理界面与CLI

**设计目标**：就是我们花大量时间设计的“数据中心”页面，它是所有数据维护工作的总入口。

#### 5.1 Web管理界面（数据中心，重构版）
整合所有子系统的全局页面，Tab结构体如下：

- **Tab 1：综合看板**
  - 数据源整体健康红绿灯。
  - 各资产类型（K线/持仓/账单/机构）完整度环形图。
  - 活动任务列表（正在跑的补数据任务）。
  - 最近告警消息流（数据缺失、解析失败、差异超限）。

- **Tab 2：行情同步（原行情数据维护）**
  - 左侧：交易所→品种→合约跟踪树。
  - 右侧：多维度卡片“日线行情 / 分钟行情”，完全复用我们上一轮设计的**差异展示列表 + 一键补齐**。

- **Tab 3：账单管理（原账单数据维护）**
  - 左侧：按月份筛选的日期列表，高亮缺失日。
  - 右侧：账单健康一览表（结算日期/上传状态/解析状态/数据质量/操作），提供**上传、重新解析、修复错误行**的入口。
  - **账单与行情对账**：此Tab下有一个子功能“成交匹配度分析”，针对已配对的账单成交和实时行情进行比对，生成差异报告（如成交价严重偏离当日价格）。

- **Tab 4：机构持仓**
  - 专门监控中金所/上期所等前20名持仓数据的下载状态，确保P1分析的底层数据完整。

#### 5.2 RESTful API & CLI
- 对外暴露所有查询、触发同步的API。
- 提供 `cli.py` 脚本，支持无头环境（服务器）下执行全量初始化、批量修复、备份恢复任务。

### 六、 前端交互：从差异到修复的闭环

通过这个架构，用户面对“不知道最新数据下载到哪一天”痛点时的操作流变为：

1.  **进入数据中心→综合看板**，发现“行情数据”卡片显示黄色警告：缺失3天分钟线。
2.  点击“行情数据”→ 进入【行情同步Tab】。
3.  在左侧树看到`CFFEX -> IM -> IM2406`亮黄灯。
4.  右侧详情表一目了然：本地分钟线最新到4月28日，理论应有到5月1日，缺失3天。
5.  点击 [补齐此合约] 或 [补齐全部缺失]。
6.  右下角日志窗口实时输出：“IM2406分钟线下载 2026-04-29...完成”，片刻后黄灯变绿，看板恢复健康，内心安稳。
