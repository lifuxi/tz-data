## 一、Celery 调度器升级：PersistentScheduler → RedBeat

### 1.1 背景

tz-data 的 Celery Beat 原使用默认的 `PersistentScheduler`（SQLite 文件持久化），存在以下问题：
- 多进程启动时产生多个 Beat 实例，导致同一任务每分钟重复调度 3 次
- 旧 `celerybeat-schedule.db` 残留导致调度记录重复
- 不支持分布式锁，无法防止多实例冲突

### 1.2 实施内容

| 文件 | 变更 |
|------|------|
| `scheduler/celery_app.py` | 新增 `beat_scheduler='redbeat.RedBeatScheduler'`、`beat_scheduler_lock_timeout=60`、`redbeat_key_prefix='redbeat:'` |
| `start.bat` | 更新注释标注 redbeat |
| `start-backend.bat` | 更新注释说明 redbeat 已自动配置 |
| `quick-start.bat` | 更新启动提示文本 |
| `C:\myspace\tz2.0\src\tasks\celery_config.py` | tz2.0 同步升级 redbeat，使用独立 Redis key 命名空间 |
| `C:\myspace\tz2.0\start-all.bat` | 更新启动提示 |
| `C:\myspace\tz2.0\start-all.ps1` | 更新启动提示 |

### 1.3 设计要点

- **Redis 隔离**：tz-data 使用 db0/db2，tz2.0 使用 db4/db3
- **分布式锁**：tz-data 使用 `celery_beat_lock`，tz2.0 使用 `tz2_beat_lock`
- **Key 前缀隔离**：tz-data `redbeat:`，tz2.0 `tz2:redbeat:`
- 无需 `--scheduler` 参数，配置自动生效

### 1.4 验证结果

- 调度频率：从 3 次/分钟降至 1 次/分钟
- 34 个定时任务全部在 RedBeat 中正常跟踪
- 无重复调度

## 四、数据质量审计引擎

### 4.1 DataQualityAuditor 统一入口

| 文件 | 行数 | 说明 |
|------|------|------|
| `verify/data_quality_auditor.py` | 735 | 统一质量审计引擎，编排多维度检查 |

支持 7 个审计维度：

| 维度 | 检查项 |
|------|--------|
| Market 数据 | 完整性、异常值、一致性、重采样验证、Schema 对齐 |
| Trading 数据 | 跨库一致性、业务规则（FIFO/PnL）、Schema 对齐 |
| Analysis 数据 | Feature/IV/Greeks 完整性、范围验证、Schema 对齐 |
| QuestDB 一致性 | SQLite vs QuestDB 数据量/内容对比 |
| 账单解析 | CFMMCParser 历史账单解析验证 |
| Celery 任务 | 模块可导入性、Beat 调度健康检查 |
| 资金对账 | bills.db 流水摘要平衡校验 |

### 4.2 CLI 入口

| 文件 | 说明 |
|------|------|
| `cli/run_quality_audit.py` | CLI 工具，支持 `--scope` 分维度审计和 `--output` JSON 导出 |

使用方式：
```bash
python -m tzdata_pkg.cli.run_quality_audit              # 全量审计
python -m tzdata_pkg.cli.run_quality_audit --scope market  # 仅市场数据
python -m tzdata_pkg.cli.run_quality_audit --scope trading # 仅交易数据
python -m tzdata_pkg.cli.run_quality_audit --scope questdb # 仅 QuestDB 一致性
python -m tzdata_pkg.cli.run_quality_audit --output report.json  # 导出 JSON
```
