# 交易日历与合约生命周期设计方案（成本与专业平衡版）

> **目标**：为个人量化交易系统设计一套**低成本、可维护、专业度足够**的交易日历与合约生命周期管理方案。优先保证中金所（CFFEX）、上期所（SHFE）等国内主流交易所的准确性，适度兼顾外盘扩展性。采用「免费数据源+人工修正」策略，避免复杂自动化。

---

## 一、设计原则

1. **数据源优先**：使用 Tushare Pro（免费积分）、AkShare 等公开数据作为基础日历来源，减少手工录入。
2. **分层存储**：核心数据（交易所日历、合约基础信息）存本地数据库，衍生数据（如主力合约连续、换月映射）通过规则生成或缓存。
3. **容错设计**：支持人工覆盖（如突发休市），并记录操作日志。
4. **性能折中**：采用预计算 + 简单缓存，避免实时扫描大量历史数据。

---

## 二、数据模型设计（轻量版）

### 2.1 交易所基础信息表
存储交易所基本信息，用于时区、起始日期等。

| 字段 | 说明 | 示例 |
|------|------|------|
| exchange | 代码 | CFFEX |
| name | 全称 | 中国金融期货交易所 |
| tz | 时区 | Asia/Shanghai |
| open_date | 数据起始日 | 2010-04-16 |
| is_active | 是否跟踪 | 1 |

### 2.2 交易日历表（核心）
存储每个交易所的每一日是否为交易日，以及类型。

| 字段 | 说明 |
|------|------|
| exchange | 交易所代码 |
| trade_date | 日期 |
| is_open | 1=全天交易，0=休市，2=半日交易 |
| pre_date | 前一个交易日（预计算） |
| next_date | 下一个交易日（预计算） |
| remark | 原因（如“春节”、“台风休市”） |

**维护方式**：
- **初始化**：从 Tushare `trade_cal` 接口批量导入所有交易所历史日历。
- **日常更新**：每日收盘后调用 Tushare 获取最新交易日（通常提前一年公布），增量插入。
- **人工修正**：提供简单 Web 界面，允许手动修改某一天的 `is_open` 和 `remark`，并记录修改日志。

### 2.3 商品（品种）信息表
记录每个可交易品种的基本属性。

| 字段 | 说明 | 示例 |
|------|------|------|
| product_id | 品种代码 | MO |
| exchange | 所属交易所 | CFFEX |
| name | 名称 | 中证1000股指期权 |
| type | 类型 | index_option / index_future / commodity_future |
| list_date | 上市日期 | 2022-07-22 |
| delist_date | 退市日期（默认 null） | - |
| multiplier | 合约乘数 | 100 |
| price_tick | 最小变动价位 | 0.2 |
| margin_rate | 保证金率（期货） | 0.12 |
| option_style | 欧式/美式 | European |

### 2.4 合约信息表
记录每个具体合约的生命周期关键日期。

| 字段 | 说明 | 示例 |
|------|------|------|
| symbol | 合约代码 | MO2506 |
| product_id | 所属品种 | MO |
| list_date | 挂牌日 | 2024-06-24 |
| last_trade_date | 最后交易日 | 2025-06-20 |
| expiry_date | 到期日（期权） | 2025-06-25 |
| delivery_date | 交割日（期货） | 2025-06-23 |
| strike | 行权价（期权） | 5500 |
| call_put | C/P | C |

**自动生成规则**：
- 对于期货，根据交易所公布的挂牌规则（如当月、下月、随后两个季月），在每月合约到期后自动生成未来一年的合约记录。规则可固化在代码中（例如 IM 期货：挂牌合约月份为当月、下月及随后两个季月；每个合约最后交易日为到期月份的第三个星期五）。
- 对于期权，先确保标的期货/指数合约存在，然后根据行权价间距规则自动生成平值附近合约（或从 Tushare `opt_basic` 同步）。**建议优先使用 Tushare 接口直接拉取合约列表**，避免自己生成复杂的行权价序列。

### 2.5 特殊日期覆盖表
用于记录规则无法自动处理的异常情况（如临时休市、保证金调整）。

| 字段 | 说明 |
|------|------|
| exchange | 交易所 |
| trade_date | 发生日期 |
| affect_products | 影响的产品（空表示所有） |
| type | 类型：休市/半日/保证金调整/最后交易日顺延 |
| value | 调整值（如保证金比例） |
| operator | 操作人（系统/管理员） |

查询时，此表数据优先级高于自动生成的日历。

## 三、维护流程（低人工成本）

### 3.1 初始数据准备
1. 首次运行系统，调用 Tushare `trade_cal` 获取所有交易所自开市至今年底的交易日历，存入 `exchange_calendar`。
2. 调用 Tushare `fut_basic` 和 `opt_basic` 获取全量期货、期权合约信息，存入 `contract` 表。
3. 手动补充或核对商品表（`product`）中的基本信息（乘数、保证金率等）。

### 3.2 每日更新任务（定时）
- **交易日历**：每天 18:00 调用 Tushare `trade_cal` 获取未来三个月的交易日数据（若尚未存在），并补充 pre_date/next_date。
- **合约信息**：每周运行一次同步任务，检查是否有新上市的合约（如当月合约到期后，下月合约开始挂牌）。调用 `fut_basic`/`opt_basic` 获取新合约，若本地不存在则插入。
- **到期标记**：每日自动检查 `last_trade_date` ≤ 当前日期的合约，将其状态标记为 `is_dead=1`，并在回测/交易中不再使用。

### 3.3 人工介入场景
- 交易所突发休市（如台风、领导人悼念日）：用户可在管理界面添加 `special_date` 记录，系统次日自动跳过该交易日。
- 保证金率调整：登录交易所官网查看公告，手动修改 `product` 表中的 `margin_rate`，并可选记录生效日期。

## 四、查询接口设计（供其他模块调用）

| 功能 | 查询方法 | 实现要点 |
|------|----------|----------|
| 判断某日是否为交易日 | `is_trading_day(exchange, date)` | 先查 `special_date` 覆盖，再查 `exchange_calendar`。 |
| 获取两个日期之间的交易日列表 | `get_trading_days(exchange, start, end)` | 使用数据库中预计算的 `pre_date/next_date` 链表，或直接范围查询 `is_open=1`。 |
| 获取某合约的剩余交易日 | `get_remaining_trading_days(contract_id, current_date)` | 从合约的 `last_trade_date` 往前数到 current_date 的交易日个数。 |
| 获取主力合约（连续） | `get_main_contract(product_id, date)` | 根据当前日期，选择持仓量最大或成交量最大的活跃合约。可每周末预计算一次并缓存。 |
| 获取合约到期日期 | `get_expiry_date(contract_id)` | 直接读取表字段。 |

**性能优化**：
- 将常用日历（如近三年交易日）缓存在内存 dict 中，避免每次查询数据库。
- 对于回测引擎，预先加载整个回测区间内的交易日列表到数组，按索引访问。

## 五、专业性与成本权衡点

| 功能点 | 专业软件做法 | 本方案简化点 | 理由（成本控制） |
|--------|--------------|--------------|------------------|
| 全球交易所覆盖 | 支持数百个交易所 | 仅国内四大期货交易所 + 上交所（若有ETF期权） | 个人主要交易国内品种 |
| 实时休市推送 | 自动接收官方消息 | 人工录入 | 临时休市一年最多一两次，手动可接受 |
| 合约自动生成 | 复杂规则引擎（连续活跃、展期） | 主要靠数据源同步，少量规则生成 | Tushare 已提供合约列表，无需自研 |
| 历史规则回溯 | 记录每次规则变更 | 不保存历史版本，用当前规则 | 个人回测只需一致性，无需精确复现历史规则变更 |
| 主力合约连续索引 | 实时计算成交/持仓量 | 每日收盘后预计算一次并缓存 | 实时计算消耗资源，日频已够用 |
| 夏令时自动转换 | 内置全球时区库 | 仅支持北京时间，外盘手动调整 | 暂不交易外盘 |

## 六、分阶段实施建议

### 阶段一：基础日历（1天）
- 部署表结构。
- 编写脚本从 Tushare 导入历史交易日历。
- 实现 `is_trading_day` 基础查询函数。
- **验证**：能正确判断 2024 年春节假期休市。

### 阶段二：合约同步（2天）
- 编写脚本同步期货、期权合约信息。
- 实现合约到期自动标记。
- **验证**：查询 `IM2506` 的 last_trade_date 与交易所公布一致。

### 阶段三：交易日预计算与缓存（1天）
- 增加 `pre_date`/`next_date` 字段并每日更新。
- 实现内存缓存，优化查询性能。
- **验证**：连续查询 1000 次日历，耗时 < 0.1 秒。

### 阶段四：主力合约与换月映射（2天）
- 编写主力合约判定逻辑（基于持仓量）。
- 生成主力连续合约映射表（用于回测）。
- **验证**：IM 主力合约在到期日前一周自动切换到次主力。

### 阶段五：管理界面（可选，2天）
- 开发简单的 Web 页面，支持查看和修改特殊日期、保证金率。
- 记录操作日志。

---

## 七、总结

本方案在 **数据源免费**、**维护工作量小**（每日自动化 + 偶尔人工）的前提下，实现了专业软件的核心功能：精确交易日判断、合约生命周期跟踪、主力合约映射。不足在于无法实时处理全球突发事件和多时区，但对个人国内量化交易已足够。未来若需扩展外盘，可再增加时区表和相应的日历来源（如 Yahoo Finance 的日历接口）。
